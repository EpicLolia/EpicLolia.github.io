条款05 了解C++默认编写并调用了哪些函数

  • 如果成员中有reference或const类型，或者一个derive dclass的base class有private赋值操作符，编译器将不会生成默认copy assignment

请记住

  • 编译器可以暗自为class创建default构造函数、copy构造函数、copy assignment操作符，以及析构函数。

条款06 若不想使用编译器自动生成的函数，就该明确拒绝

请记住

  • 为驳回编译器自动（暗自）提供的机能，可将相应的成员函数声明为private并且不予实现。
  • 使用像Uncopyable这样的base class 也是一种做法。

条款07 为多态基类声明virtual析构函数

  • virtual机制会给类增加一个指针的内存，同时也失去了可移植性（其他语言）

  • 并不是作为基类就要virtual析构函数，只有涉及多态，使用基类指针的时候，会有derived class不析构的问题。

  • 如果将析构函数声明为pure-virtual，也请给出其定义，因为它是一定会被调用到的，不写会报link错误。

请记住

  • polymorphic（带多态性质的）base classes应该声明一个virtual析构函数。如果class带有任何virtual函数，他就应该拥有一个virtual析构函数
  • Classes 的设计目的如果不是作为base class 使用，或不是为了具备多态性（polymorphic），就不该声明virtual析构函数
  • PS: virtual == 多态

条款08 别让异常逃离析构函数

请记住

  • 析构函数绝对不要突出异常。如果一个被析构函数调用的函数可能抛出异常，析构函数应该捕捉任何异常，然后吞下他们（不传播）或结束程序。
  • 如果客户端需要对某个操作函数运行期间抛出的异常做出反应，那么class应该提供一个普通函数（而非在析构函数中）执行该操作。

条款09 绝不在构造和析构过程中调用virtual函数

  • 这是一个与java和C#不同的地方

请记住

  • 在构造和析构期间不要调用virtual 函数，因为这类调用从不下降至derived class（比起当前执行构造和析构函数的那层）

  • 三种方法保证赋值安全

    • 调整语序，确保在delete前完成必要的操作

【Effective C++】1 让自己习惯C++

发表于 2022-10-20 分类于 编程语言 ， C/C++ ， Effective C++

条款01 视C++为一个语言联邦

4个主要次语言：

  • C
  • Object-Oriented C++
  • Template C++
  • STL

请记住：

  • C++ 高效编程守则视状况而变化，取决于你使用C++的哪一部分

条款02 尽量以 const，enum，inline 替换 #define

对于定义常量的宏，可以使用enum或const变量来代替。其中需要注意的是enum不可以取地址，更像define，而const是可以取址的。

对于定义函数的宏，可以用 template inline 来代替 define 函数。

请记住：

  • 对于单纯常量，最好以从const对象或enums替换#define
  • 对于形似函数的宏(macros)，最好使用inline函数替换#define

条款03 尽可能使用const

首先明确：const是一条由编译器来帮助你完成的约束！

几种使用const的常见情景

  1. 对于指针，const在*左边代表指向常量，在*右边代表指针是常量

  1. 对于迭代器

    • const_iterator 代表指向常量，不可以修改内容
    • const iterator代表是常迭代器，不可以改变指向的对象(如 ++iterator)

  2. 对于函数的声明

    • 使用const返回值，可以有效避免”想要输入==却输入=“的情况（编译器将会直接报错）
    • 函数的const参数也是如此，可以防止函数意外修改参数的内容

  1. 对于const成员函数

    • 可以清晰知道哪个函数改变类的成员

    • 确保const对象可以使用此函数，使“操作const对象”成为可能

修改const对象

​ 即便是常对象，还是有手段改变其内容的，比如成员指针指向的内容，就不属于对象本身（bitwise constness），可以进行修改。

​ 也可以直接声明mutable成员，这样const函数也可以在客户端不知情（logical constness）的情况下修改成员变量。也就是说，const对象只保证对象内存不被修改，成员指针变量不会变化，但是成员指针指向的内存不归const对象管。有需要的话，这由程序员自己来保障（如给const函数声明const返回值）。

关于const成员函数的小技巧

​ 如果non-const和const版本的成员函数功能一致，那么为一个成员函数声明两个形态会使代码量爆炸，可以使用non-const调用const。

​ 如果反过来用const调用non-const是违反const成员函数的宗旨的。你承诺过不改变类的状态，但是non-const函数有可能改变类的状态。这么做需要使用const_cast将*this的const解掉，这是大难临头的前兆。

请记住：

  • 将某些东西声明为const可以帮助编译器侦测出错误用法。const可被施加于任何作用域内的对象、函数参数、函数返回类型、成员函数本体。
  • 编译器强制实施bitwise constness，但你编写程序时应该使用“概念上的常量性”（conceptual constness）
  • 当const 和non-const 成员函数有着实质等价的实现时，令non-const版本调用const版本可以避免代码重复。

条款04 确定变量在被使用前已先被初始化

对于内置类型（int, char，pointer等）

  1. 全局、静态变量会默认初始化内存为0，当然也可以进行显式初始化。

  2. 如果声明在堆、栈空间，就需要手动初始化。对于其他类型，由构造进行初始化，确保将每一个成员都初始化。

  • 不要混淆赋值（assignment）和初始化（initialization）。在构造函数主体里面进行的是赋值，而非初始化。这种情况下非内置类型成员（如：string）会先执行自己的默认（default）构造函数进行初始化，再进行赋值。应对非内置类型成员使用列表初始化（member initialization list），效率更高，根本原因就是直接调用了成员的有参构造函数。当然，对于内置类型区别不大（两种方式成本相当），为了一致性，我们也将它写在列表中初始化。最好的做法就是在初值列中列出所有成员。

  • const 和 references 一定需要初值，不能赋值。

  • 对于多份构造函数，都写列表初始化会显得冗长，可以将“赋值表现得像初始化一样好”的成员变量移到一个private函数中，供所有构造函数调用，但最好不要这么做。

  • 列表初始化是按照声明顺序进行的，与列表顺序无关！最好写成一样的顺序。小心有成员变量使用未赋初值的成员进行初始化。

  • 对于static对象，其寿命从构造开始到程序结束（main()结束）为止。函数内的叫local-static对象，其他叫non-local static对象（global, namespace, class, file）。编译单元 （translation unit）是指产出单一目标文件（single object file）的源码。

    • 如果一个non-local static对象的初始化操作使用了另一个不同编译单元中的non-local static对象，后者可能还未初始化！！C++对此无明确定义。
    • 普通的多文件全局变量就是如此（extern声明的变量）。
    • 当然 static 和 extern 是冲突的！hhhh
    • 一个好的解决办法就是使用单例（Singleton）模式，将变量用函数包起来，转化成local static对象，函数返回其引用。
    • local static对象会在第一次遇到函数定义式（调用函数）时初始化。如果不调用就不构造！性能极佳！
    • 这样的函数是inline的绝佳模板。当然inline不会使static对象拷贝多份， 代码拷贝不⼀定是c代码的拷贝，可以是编译后的汇编代码的拷贝。而函数在代码段，变量则在数据段。拷贝的只是函数部分。静态变量在函数部分保存的仅仅是该变量的地址。static变量不是放在栈⾥的,是放在静态数据区.和全局变量存放在⼀起.不能将它和局部变量混为⼀谈。——取自网络回答
    • 这样的简单的单例方法是线程不安全的。任何一种non-const static在多线程环境下“等待某事发生”都是不安全的，可能发生初始化的竞速形势（race conditions）。可以在程序单线程启动阶段（single_threaded started portion）手动把所有单例函数调用一遍。
      • 注意：在Ｃ＋＋１１中这个问题解决了
    • 当然这种方法也解决不了AB相互初始化的问题，但这种坏情况也只能说是自作自受。hhh

请记住：

  • 为内置对象进行手工初始化，因为C++不保证初始化它们。
  • 构造函数最好使用成员初值列（member initializatiion list），而不要在构造函数本体中使用赋值操作，初值列列出的成员变量，其排列顺序应该和声明顺序一致。
  • 为免除“跨编译单元之初始化次序”问题，请以local static对象替代non-static对象。

【Effective C++】3 资源管理

发表于 2022-10-20 分类于 编程语言 ， C/C++ ， Effective C++

条款13 以对象管理资源

  • 智能指针默认只会delete一个元素（而非delete[]），如果用来管理数组的话就需要自行定义删除器。

请记住

  • 为防止资源泄露，请使用RAII对象，他们在构造函数中获得资源并在析构函数中释放资源。
  • 两个常常被用的RAII class是shared_ptr和auto_ptr .前者通常是较佳的选择，因为其copy行为比较直观。若选择auto_ptr，复制动作会使它（被复制物）指向null。
  • PS: C++11 推出了shared_ptr, unique_ptr, weak_ptr，请不要再使用auto_ptr。

条款14 在资源管理类中小心copying行为

请记住

  • 复制RAII 对象必须一并复制它所管理的资源，所以资源的copying行为决定RAII 对象的copying行为。
  • 普遍而常见的RAII class copying 行为是：抑制copying、实行引用计数法。不过其他的行为也都有可能被实现。

条款15 在资源管理类中提供对原始资源的访问

  • operator 隐式转换

请记住

  • APIs 往往要求访问原始资源（raw resources），所以每一个RAII class 应该提供一个“取得其所管理之资源”的方法　　　　　　　　　　　　　　　　　
  • 对原始资源的访问可能经由显示转换或隐式转换。一般而言显式转换比较安全，但隐式转换对客户比较方便。

条款16 成对使用new和delete时要采取相同形式

请记住

  • 如果你在new表达式中使用 [ ] ，必须在相应的delete表达式中也使用 [ ] 。如果你在new表达式中不使用 [ ] ，一定不要在相应的delete表达式中使用 [ ]。

条款17 以独立语句将newed对象置入智能指针

请记住

  • 以独立语句将newed对象储存于（置入）智能指针内。如果不这样做，一旦异常抛出，有可能导致难以察觉的资源泄露。

【Effective C++】5 实现

发表于 2022-10-20 分类于 编程语言 ， C/C++ ， Effective C++

条款26 尽可能延后变量定义式的出现时间

  • 尽可能延后变量的定义式直到你可以清楚地赋予其初值，即调用有参构造函数，而不是先构造后赋值。

  • 对于循环地问题

    • 在循环体外定义变量：1个构造函数+1个析构函数+n个赋值操作 （弊端：变量会覆盖整个循环）
    • 在循环体内定义变量：n个构造函数+n个析构函数
    • 一般请用第二种，除非这段代码是效率高度敏感的，并且变量的赋值操作成本低于 构造+析构。

请记住

  • 尽可能延后变量的定义式的出现。这样做可以增加程序的清晰度并改善程序的效率。

条款27 尽量少做转型动作

  • 旧式转型 old-style-casts
    1. (T)expression // C风格
    2. T(expression) // 函数风格
    • 如今唯一使用旧式转型的地方，可能就是explicit构造函数的参数转换了（只是因为看起来顺眼hhh），请尽可能使用新式转换！！！

  • 新式转型 new-style or C++-style casts

    1. const_cast ( expression ) // 将常量性擦除
    2. dynamic_cast ( expression ) // 在继承体系中“安全向下转型”，向子类转型
      • 可以考虑使用virtual来避免类的转型，将函数往继承体系上方移动
      • 可以考虑直接使用derived class的指针，“使用类型安全的容器”
    1. reinterpret_cast ( expression ) // 低级转型，实际动作可能取决于编译器，不可移植
    2. static_cast (expression ) // 强迫隐式转换
  • 新式转型返回的值可能是一个副本！！对其操作不会影响本体。
  • 如果你发现你打算转型，那活脱是一个警告信号，尤其是dynamic_cast。
  • 对象的布局方式和地址计算方式随编译器的不同而不同！！！不要以此进行转型！！（比如：把对象转成char* 妄图对其进行逐字节修改）

请记住

  • 如果可以，尽量避免转型，特别是在注重效率的代码中避免dynamic_casts。如果有个设计需要转型动作，试着发展为无需转型的替代设计。
  • 如果转型是必要的，试着将他们隐藏于某个函数背后。客户随后可以调用该函数，而不需要将转型放进他们自己的代码中。
  • 宁可使用C++-style（新式）转型，不要使用旧式转型。前者很容易辨认出来，而且也比较有着分门别类的执掌。

条款28 避免返回handles指向对象内部成分

  • 但返回handle这一做法本身，就要面临“handle比其所指对象更长寿”的风险

请记住

  • 避免返回handles（包括reference、指针、迭代器）指向对象内部。遵守这个条款可增加封装性，帮助const成员函数的行为像一个const，并将发生“虚吊号码牌”（dangling handles）的可能性降至最低。

条款29 为“异常安全”而努力是值得的

  • 不要为了表示某件事发生而改变对象状态，除非它真的发生了。（即应该在事情确实发生之后改变对象状态，事件可能因为异常失败）

  • 当异常抛出时，异常安全的函数会：

    • 不泄露任何资源
      • 可以以对象管理资源（条款13），shared_ptr
    • 不允许数据败坏（比如野指针）

  • 异常安全函数的三个保证：

    • 基本承诺：如果异常被抛出，程序内的任何事物仍然保持在有效状态之下。
      • 至少要保证这一点
    • 强烈保证：如果异常被抛出，程序状态不会改变。
      • copy and swap策列：在一个函数中先对副本进行操作，副本操作无误后再通过swap交换。（swap保证nothrow）
      • 但强烈保证往往也伴随着时间和空间消耗
    • 不抛掷（nothrow）保证：承诺绝不抛出异常，因为他们总是能完成他们原先承诺的功能
      • 这个很难做到
  • 如果函数内使用了异常不安全的函数，那么这个函数也是异常不安全的。

请记住

  • 异常安全函数（Exception-safe functions）即使发生异常也不会泄露资源或允许任何数据结构败坏。这样的函数区分为三种可能的保证：基本型、强烈型、不抛异常型
  • “强烈保证”往往能够以copy-and-swap 的策列实现，但“强烈保证”并非对所有函数都可实现或具备现实意义
  • 函数提供的“异常安全保证”通常最高只等于其所调用之各个函数的“异常安全保证”中的最弱者。

条款30 透彻了解inlining的里里外外

  • 编译器往往只优化“不含函数调用”的代码，inline可以使编译器对函数也结合上下文进行优化。

  • inline可能导致代码膨胀，造成程序体积膨胀。

  • inline操作是在调用的时候进行的，会不会inline由编译器决定。如果使用函数指针调用inline函数，那么这个函数 或许 不会执行inline操作。

  • inline之后的函数往往也是无法调试的，很难在一个没有本体的函数里打断点。（debug模式下可能并没有执行inline）

  • inline只是对编译器的一个申请，不是强制。inline一般是编译期行为，所以一般置于头文件中（template也是这样）。

    • 隐喻方式：将函数定义在class定义式内
    • 明确声明：在定义式前加inline。

  • inline函数的更改也会导致其用户程序必须重新编译！！ 而non-inline函数只要重新连接就好了，如果库采用动态连接，那么甚至可以神不知鬼不觉地更改函数。

请记住

  • 请将大多数inlining 限制在小型、被频繁调用的函数身上。这可使日后的调试过程和二进制升级过程（binary upgradability）更容易，也可使潜在的代码膨胀问题最小化，使程序的速度提升最大化。
  • 不要只因为function template 出现在头文件，就将他们声明为inline。

条款31 将文件间的编译依存关系降至最低 (看不懂

请记住

  • 支持“编译依存性最小化”的一般构想是：相依于声明式，不要相依于定义式。基于此构想的两个手段是Handle class 和Interface classes。
  • 程序库头文件应该以“完全仅有声明式”（full and declaration-only forms）的形式存在。这种做法不论是否涉及templates 都适用。

【Effective C++】6 继承与面向对象设计

发表于 2022-10-20 分类于 编程语言 ， C/C++ ， Effective C++

条款32 确定你的public 继承塑模出 is-a 关系

  • 若class D（Derived） public 继承 class B（Base），那么B对象能派上用场的地方，D一样也可以（Liskov Substitution Principle）

请记住

  • “public继承”意味着is-a。适用于base classes身上的每一件事情一定也适用于derived classes身上，因为每一个derived class 对象也都是一个base class 对象。

条款33 避免遮掩继承而来的名称

  • 即便是覆盖虚函数，它还是会遮盖同名函数。

请记住

条款39 明智而审慎地使用private 继承

  • private继承只继承实现，不继承接口，基类的public 和protected 都会变成private，是一个is-implemented-in-terms-of。
  • 尽可能使用复合（或者继承+复合，有必要的话，即当需要覆写virtual），必要时才使用private继承（当protected成员和virtual函数牵扯进来）
  • 继承是传递的，一个类private继承了某个功能类，那么这个类的派生类也有能力覆写virtual函数，这往往是不合理的。（C++11有final，可以实现了）
  • private继承一个不占内存的类（empty base），可以实现对象尺寸最小化。

请记住

  • private 继承意味着 is-implemented-in-terms-of（根据某物实现出）。它通常比复合（composition）的级别低。但是当derived class 需要访问protected base class 的成员，或需要重新定义继承而来的virtual 函数时，这么设计是合理的。
  • 和复合（composition）不同，private 继承可以造成empty base 最优化。这对于致力于“对象尺寸最小化”的程序库开发者而言，可能很重要。

条款40 明智而审慎地使用多重继承

  • 对于“钻石型”继承，编译器往往默认会生成多份重复的基类的成员。如果要避免，就需要对重复的基类使用虚继承（成为虚基类），但这会消耗更多性能。

  • virtual base的初始化责任是由继承体系中最底层（most derived）class负责！

  • 使用virtual base的忠告：

    • 别用
    • 避免在virtual base 里放数据
  • private继承可以合理地被多重使用，该用就用不要害怕hhh。

请记住

  • 多重继承比单一继承复杂。他可能导致新的歧义性，以及对virtual继承的需要。
  • virtual继承会增加大小、速度、初始化（及赋值）复杂度等等成本。如果virtual base class 不带任何数据，将是最具实用价值的情况。
  • 多重继承的确有正当用途。其中一个情节涉及“public 继承某个 Interface class”和“private继承某个协助实现的class”两相组合。

条款41 了解隐式接口和编译期多态

  • 显式接口（explicit interface），通常由函数签名式（名称、参数类型、返回类型）构成
  • 隐式接口（implicit interface），由有效表达式（valid expressions）组成。即在模板中使用的需要在编译期间确定的函数（这是我自己的理解）。

  • 对classes而言接口是显式的（explicit），以函数签名为中心。多态则是通过virtual函数发生于运行期。
  • 对template参数而言，接口都是隐式的（implicit），奠基于有效表达式。多态则是通过template具现化和函数重载解析（function overloading resolution）发生于编译期

条款46 需要类型转换时请为模板定义非成员函数

  • 使用条款24的方式不能解决 模板类的隐式转换。函数模板依赖实参推导！函数模板根据调用的实参推导模板参数类型，在推导过程中不会采纳“通过构造函数而发生”的隐式转换。即函数实参的隐式转换必须以函数具现化为前提！但是模板函数又需要具体的（准确的）实参类型来具象化函数。
  • class template（类模板）不依赖实参推导，所以可以将non-member函数声明为类的friend函数。所以当类具象的时候，模板friend函数也具象化了。又因模板的声明和定义不好分开定义，链接问题比较复杂（声明具象化的时候找不到对应的具象化的定义），我们将声明和定义写到一起。
  • 有趣的是，我们在这里使用friend 并不是为了函数能获得类中的访问权限，而是为了模板函数能伴随模板类自动具现化。

请记住

  • 当我们编写一个class template，而它所提供之“与此template相关的”函数支持“所有参数之隐式类型转换”时，请将那些函数定义成“class template 内部的friend 函数”。

条款46 需要类型转换时请为模板定义非成员函数

  • 使用条款24的方式不能解决 模板类的隐式转换。函数模板依赖实参推导！函数模板根据调用的实参推导模板参数类型，在推导过程中不会采纳“通过构造函数而发生”的隐式转换。即函数实参的隐式转换必须以函数具现化为前提！但是模板函数又需要具体的（准确的）实参类型来具象化函数。
  • class template（类模板）不依赖实参推导，所以可以将non-member函数声明为类的friend函数。所以当类具象的时候，模板friend函数也具象化了。又因模板的声明和定义不好分开定义，链接问题比较复杂（声明具象化的时候找不到对应的具象化的定义），我们将声明和定义写到一起。
  • 有趣的是，我们在这里使用friend 并不是为了函数能获得类中的访问权限，而是为了模板函数能伴随模板类自动具现化。

请记住

  • 当我们编写一个class template，而它所提供之“与此template相关的”函数支持“所有参数之隐式类型转换”时，请将那些函数定义成“class template 内部的friend 函数”。

条款47 （看不懂）请使用traits（特征） classes 表现类型信息

请记住

  • Traits classes 使得“类型相关信息”在编译期可用。它们以templates 和 “templates 特化”完成实现。
  • 整合重载技术（overloading）后，traits classes 有可能在编译期对类型执行if…else 测试。

条款48 认识template 元编程

  • Template metaprogramming（TMP，模板元编程），将运行期的工作转移到编译期。

  • 三个应用场景（举例）

    • 确保度量单位正确。TMP可以确保程序所有度量单位组合的正确性（在编译期）。
    • 优化矩阵运算。
    • 可以生成客户定制之设计模式（custom design pattern）实现品。如Strategy、Observer、Visitor等都可以多种方式实现。

请记住

  • Template metaprogramming 可将工作由运行期转移到编译期，因而得以实现早期错误侦测和更高的执行效率。
  • TMP 可被用来生成“基于政策选择组合”（based on combinations of policy choices）的客户定制代码，也可用来避免生成对某些特殊类型并不适合的代码。

条款53 不要轻忽编译器的警告

请记住

  • 严肃对待编译器发出的警告信息。努力在你的编译器的最高（最严苛）警告级别下争取“无任何警告”的荣誉
  • 不要过度依赖编译器的报警能力，因为不同的编译器对待事情的态度并不相同。一旦移植到另一个编译器上，你原本依赖的警告信息有可能消失。
