用 Lint 代替提示词：中文 Markdown 排版的程序化约束

发表于 2026-05-13 更新于 2026-05-17 分类于 misc 阅读次数： 本文字数： 1.4k 阅读时长 ≈ 5 分钟

中文技术博客中存在大量细碎但影响阅读体验的排版问题：中英文之间缺少空格、标点前后多余空白、数字与汉字紧邻。这些问题人工写作时经常出现，AI 辅助创作也无法避免。常见的应对方式是在提示词里写排版规范，但效果并不稳定——AI 不会”记住”规则，人自己写的时候也会犯。

比起用自然语言描述规范，通过 lint 进行程序化约束更可靠。本文介绍一套基于 ESLint 的方案：编写自定义规则覆盖中文排版检查，并通过 Claude Code Hook 和 VS Code 保存动作实现自动修复，让 AI 和人共享同一套规则。

使用 ESLint 规范 Markdown

Markdown lint 工具不止一种（如 markdownlint），本文选择 ESLint 的理由是：ESLint 更具通用性：一条工具链即可覆盖项目中的 JS、JSON、Markdown 等多种格式，减少工具碎片化。

@eslint/markdown 插件提供了 GFM 解析器和推荐规则集，支持 Front Matter 解析（标题、标签等同样需要排版检查）和 LaTeX 数学公式（避免 $...$ 被误报）。更关键的是，ESLint 支持编写自定义规则——这意味着可以把中文排版规范程序化为可执行的检查逻辑。

用自定义规则优化中文排版

现有 Markdown lint 规则面向英文设计，对中文排版几乎没有覆盖。以下是中文文档中最常见的排版问题：

问题类型 反例 正例
中英文之间缺少空格 你好world 你好 world
中文与数字之间缺少空格 共100个 共 100 个
中文标点前后多余空格 你好 ，世界 你好，世界
货币/百分号与数字间多余空格 $ 100 $100

这些问题有一个共同特点：规则明确、可用正则描述、修复方式确定。非常适合用 lint 规则自动化处理。

本项目的两条规则

基于上述机制，本项目实现了两条规则：

cjk-spacing：处理 CJK 字符与 Latin 字符的间距。CJK 紧邻 Latin 时自动插入空格，CJK 标点前后出现多余空格时自动移除。

number-spacing：处理数字与 CJK 字符的间距。这里的”数字”不仅是 \d+，还涵盖货币前缀（$100）、百分号后缀（50%）、千分位（1,000,000）和小数（3.14）——这些边界情况正是提示词难以精确描述的。

集成到 AI + 人的工作流

规则写好后，下一步是让 AI 和人在编辑文档时自动执行检查。

Claude Code Hook：AI 编辑后自动修复

Claude Code 的 Hook 机制允许在特定事件触发时执行自定义脚本。在 .claude/settings.json 中注册一个 PostToolUse Hook，当 Claude Code 执行 Edit 或 Write 工具后自动触发 lint：

Hook 脚本从 stdin 读取工具调用信息，提取文件路径，执行 eslint --fix。可修复问题当场修复；不可修复问题通过 hookSpecificOutput 反馈给 AI，AI 据此调整后续输出。

总结

程序化约束优于提示词约束。 提示词是概率性的，lint 是确定性的。在 AI 协作的工作流中，lint 提供了一道不依赖”理解”和”记忆”的质检关卡。

本文的方案分为三层：用 ESLint 作为基础设施，用自定义规则覆盖中文排版的高频问题，用 Hook 和编辑器动作让 AI 和人两端自动执行。如果你的项目也有类似需求，可以从这两条规则出发，逐步扩展——全角/半角标点统一、链接文本格式、术语拼写检查等，都适合用同样的模式实现。规则一旦写好，就会在每次编辑中持续发挥作用。
